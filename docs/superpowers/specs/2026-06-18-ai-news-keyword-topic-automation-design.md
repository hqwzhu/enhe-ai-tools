# AI资讯关键词与专题自动化设计

## 目标

为 ENHE AI 的 `AI前沿资讯` 页面建立一套可持续演进的 `热门关键词` 与 `专题合集` 自动生成机制，替换当前写死在前端代码中的静态内容，让页面既满足站内运营可控性，也满足 SEO 增长需求。

本次设计同时覆盖两个阶段：

1. 第一阶段：基于站内数据完成自动生成、SEO 规则过滤、后台干预和前台展示。
2. 第二阶段：引入外部 SEO 数据，对第一阶段结果做加权增强，而不是推翻站内规则。

## 当前现状

- `热门关键词` 和 `专题合集` 当前是写死在 `src/app/ai-news/page-shell.tsx` 的本地常量。
- AI 资讯文章、资讯分类、资讯标签是动态数据，来自 `src/lib/public-content.ts` 中的 Prisma 查询。
- 项目已有通用埋点表 `AnalyticsEvent`，但没有面向 AI资讯搜索词的专用聚合逻辑。
- 当前没有外部 SEO 数据源接入能力，也没有关键词榜单或专题榜单的预计算层。

## 用户确认的约束

- `热门关键词` 来源包含：
  - 资讯文章标签
  - 资讯文章 `keywords / seoKeywords`
  - 资讯页站内搜索词
  - 文章热度信号
- `热门关键词` 允许后台人工干预：
  - 置顶
  - 屏蔽
  - 改名
  - 调整排序权重
- `专题合集` 第一版走 `完全自动生成`
- `专题合集` 展示数量固定为 `5`
- `热门关键词` 必须满足 SEO 条件，不能只按热度输出
- 外部 SEO 数据采用分阶段混合方案：
  - 第一阶段先用站内规则上线
  - 第二阶段再接入外部关键词/趋势数据

## 设计原则

1. 站内规则优先于外部流量数据。
2. 关键词和专题必须有内容承接，不能生成“空 SEO 页面”。
3. 自动生成必须允许人工校正，避免脏词、泛词、品牌噪声词进入前台。
4. 第一阶段上线后即使没有外部 SEO 供应商账号，也能独立运行。
5. 第二阶段以“加权增强”为目标，不改变第一阶段的数据边界和展示接口。

## 方案对比

### 方案 A：严格分两次开发

- 第一次只做站内自动生成
- 第二次再补外部 SEO

优点：

- 风险最低
- 开发节奏清晰

缺点：

- 两次改动会重复穿过同一批页面与服务层
- 第二次上线前，关键词质量上限受站内样本量限制

### 方案 B：真正一步到位，第一版就接完整外部 SEO

- 本次同时完成站内自动生成与外部 SEO 加权

优点：

- 理论上一次上线达到最终形态

缺点：

- 必须先确定外部数据源、账号、配额、请求频率、失败兜底
- 风险主要不在代码，而在第三方服务依赖

### 方案 C：一次完成双阶段架构，外部数据做成可插拔增强层（推荐）

- 本次一次性完成：
  - 站内自动生成
  - 后台干预
  - 前台替换
  - 关键词/专题评分引擎
  - 外部 SEO Provider 接口
  - 外部数据融合点
- 如果暂时没有外部数据源，就以 `No-op Provider` 运行
- 后续只需补某一个 Provider 的接入实现即可开启第二阶段

优点：

- 工程结构一步成型
- 站内版本可立即上线
- 不被第三方数据源的采购或账号准备卡住

缺点：

- 本次设计和实现会比单纯第一阶段更完整

本设计采用 `方案 C`。

## 总体架构

新增三层能力：

1. `信号采集层`
   - 采集文章标签、关键词、文章热度、站内资讯搜索词
   - 第二阶段补充外部 SEO 信号

2. `聚合评分层`
   - 关键词候选生成
   - SEO 准入过滤
   - 关键词综合评分
   - 专题聚类与专题评分

3. `运营控制层`
   - 后台干预规则表
   - 前台优先使用人工干预后的结果

前台只读取最终结果，不关心“这个词是自动算出来的还是人工修过的”。

## 热门关键词设计

### 候选来源

第一阶段候选来自四类站内信号：

1. 资讯标签 `NewsTag.name`
2. 文章关键词字段
   - `NewsArticle.keywords`
   - `NewsArticle.seoKeywords`
   - 英文页可扩展到 `englishKeywords / englishSeoKeywords`
3. AI资讯页搜索词
   - 从 `AnalyticsEvent` 中读取 AI资讯搜索事件的 query
4. 文章热度信号
   - `viewCount`
   - `isPinned`
   - `isFeatured`
   - `publishedAt` 新鲜度

### SEO 准入规则

关键词只有同时满足以下条件时，才允许进入热门关键词榜单：

1. `可读`
   - 非空
   - 去空格后长度在合理范围内
   - 过滤乱码、纯符号、纯数字、停用词

2. `有明确意图`
   - 不能是过泛词，例如“AI”“工具”“教程”这类无明确落地意图的词
   - 不能是过宽泛品牌前缀或语义不完整碎片

3. `有站内承接`
   - 用该词搜索 AI资讯页，必须能返回足够数量的相关文章

4. `有信号强度`
   - 至少满足以下之一：
     - 相关文章数量达到阈值
     - 近 30 天站内搜索次数达到阈值
     - 该词关联文章热度达到阈值

5. `可索引`
   - 必须能稳定映射到站内资讯列表查询入口

### 推荐默认阈值

- 关键词长度：`2-24`
- 查询结果数：`>= 3`
- 至少满足以下之一：
  - 关联文章数 `>= 2`
  - 近 30 天搜索次数 `>= 5`
  - 热度分达到内部下限

这些阈值做成服务层常量，后续如有必要再转后台配置，不在本次首版过度设计。

### 综合评分

通过 SEO 准入后，再计算关键词综合分：

- 标签信号分
- 关键词字段命中分
- 搜索频次分
- 文章浏览热度分
- 新鲜度分
- 人工干预加权分

最终排序逻辑：

1. 先按人工置顶分组
2. 再按综合分排序
3. 屏蔽词直接过滤
4. 改名词在最终输出阶段替换展示名

### 后台干预

第一阶段保留后台干预能力，但只针对 `热门关键词`：

- `置顶`
- `屏蔽`
- `改名`
- `手动排序权重`

后台干预不直接写死前台内容，而是作为自动结果的覆盖层。

## 专题合集设计

### 核心原则

- 完全自动生成
- 固定展示 `5` 个
- 不提供人工手填专题入口

### 生成逻辑

专题不是人工命名的页面块，而是从高质量关键词中自动聚合出来的查询主题。

候选主题来源：

- 通过 SEO 准入的关键词
- 权重较高的资讯标签
- 高频站内搜索词

聚类规则：

1. 先做关键词归一化、别名合并和去重
2. 将近义主题合并到统一 topic key
3. 以 topic key 统计：
   - 相关文章数
   - 搜索频次
   - 累计浏览量
   - 内容新鲜度

### 专题评分

专题分由以下信号构成：

- 主题下文章数
- 主题下文章累计浏览量
- 主题近 30 天搜索频次
- 最近发布时间权重
- 是否命中高质量标签

### 专题展示

- 固定输出前 `5` 个专题
- 点击后跳转到 AI资讯列表页，带主题 query
- 如果某专题候选词是中文页主题，则中文页优先展示中文标题
- 英文页使用对应英文主题名；若没有稳定英文映射，则保留统一英文规则名

### 兜底规则

如果通过评分的主题不足 `5` 个：

- 使用通过 SEO 准入的高质量标签主题补位
- 仍然不开放人工手填

## 搜索词采集设计

### 为什么需要补埋点

虽然项目已有 `AnalyticsEvent`，但当前没有“AI资讯搜索 query 可聚合读取”的稳定事件模型，因此需要为 AI资讯搜索显式记录事件。

### 第一阶段记录内容

每次在 AI资讯页提交筛选/搜索时，记录：

- `eventName`
- `path`
- `locale`
- `query`
- `category`
- `tag`
- `sort`

### 聚合窗口

第一阶段默认统计 `近 30 天` 搜索词。

## 第二阶段：外部 SEO 数据增强

### 边界

第二阶段不是“重新发明榜单”，而是给第一阶段已有结果加一层外部信号。

### Provider 抽象

引入 `ExternalSeoProvider` 接口，输出统一结构，例如：

- keyword
- externalSearchVolume
- externalTrendScore
- externalCompetitionScore
- source
- fetchedAt

### 第一版实现要求

即使本次还没有选定正式供应商，也要完成：

- Provider 接口定义
- No-op Provider
- 外部数据融合点
- 配置开关

这样后续只需补一个供应商实现，就能开启第二阶段。

### 融合策略

外部 SEO 数据只能加权，不直接越权生成词：

- 只有先通过第一阶段站内 SEO 准入的关键词，才允许吃到外部加权
- 外部数据不能把无内容承接的词强行推上前台

## 数据模型建议

### 新增：关键词人工干预表

用途：

- 存储置顶/屏蔽/改名/排序加权

建议字段：

- keyword
- locale
- isPinned
- isHidden
- displayName
- weightBoost
- createdAt
- updatedAt

### 可选新增：外部 SEO 快照表

如果第二阶段直接落地真实 Provider，则建议增加快照表，缓存外部 SEO 数据，避免每次前台请求都打第三方接口。

如果本次只完成接口与 No-op Provider，则该表可以在实现计划中评估是否同步落地。

## 服务层设计

建议新增或扩展以下服务职责：

- 读取 AI资讯搜索词聚合
- 构建关键词候选池
- 执行 SEO 准入过滤
- 计算关键词综合分
- 计算专题聚类与专题分
- 应用后台干预规则
- 为前台返回最终结果

前台 `AI前沿资讯` 页只消费：

- `keywordCloudItems`
- `topicCollectionItems`

不再持有写死常量。

## 前台行为

AI资讯页右侧栏改为读取服务端聚合结果：

- `热门关键词`
  - 展示 `10-12` 个
  - 使用最终展示名
  - 点击进入 AI资讯查询结果页

- `专题合集`
  - 展示固定 `5` 个
  - 标题来自自动主题名
  - 点击进入对应专题查询页

## 管理后台行为

第一阶段后台只增加 `热门关键词干预` 管理，不增加 `专题合集` 管理。

后台提供：

- 查看当前自动候选关键词
- 手动置顶
- 屏蔽
- 改展示名
- 调整权重

## 测试要求

至少覆盖以下测试面：

1. 关键词归一化与去重
2. SEO 准入规则
3. 关键词综合分排序
4. 人工干预覆盖逻辑
5. 专题自动聚类与 `5` 个输出规则
6. 搜索词埋点写入
7. 前台 AI资讯页不再依赖写死常量

## 风险与判断

### 可以一步到位完成两个阶段吗

可以，但前提是“两个阶段”的定义要拆开：

- `工程上一步到位`：可以，本次一次性把整体架构、自动生成、后台干预、Provider 接口全部建好。
- `业务上一步到位`：只有在已经确定外部 SEO 数据源、认证方式、额度限制的前提下才值得直接做完。

因此，本设计推荐的“一步到位”定义是：

> 本次一次性完成双阶段架构与第一阶段完整功能，并把第二阶段的外部 SEO 扩展点也一并落地；如果外部供应商条件已齐备，则可顺手完成一个正式 Provider；如果条件未齐备，则第二阶段先以 No-op Provider 占位。

这比“先做一半再返工”更稳，也比“硬接一个还没定好的第三方”更可控。

## 推荐结论

推荐本次按以下目标执行：

1. 一次完成双阶段架构
2. 一次上线第一阶段全部功能
3. 同时落地外部 SEO Provider 抽象
4. 是否真正接通第二阶段的外部供应商，取决于外部数据源是否已确定

如果外部数据源尚未确定，本次仍然值得开发，因为不会浪费；后续只需补 Provider 实现。
