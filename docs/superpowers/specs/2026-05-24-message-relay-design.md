# 传话功能设计

## 背景

在现有 ENHE AI 工具站中新增一个合规的“传话功能”。该功能不提供非实名号码、虚拟号批量代发、绕过平台限制或绕过实名的能力。它面向平台客服场景：客服基于明确业务依据创建传话工单，把消息和图片交给系统，由系统生成可审计、可过期、可受控访问的传话页面，并可通过合规短信服务发送提醒链接。

第一版目标是解决“客服需要联系用户，但不暴露双方真实手机号”的问题。系统保留业务授权依据、发送内容、图片、访问日志、风控命中和管理员操作记录。

## 产品目标

- 客服可以基于工单向用户传递文字和图片。
- 用户通过安全链接查看传话内容，不需要注册登录。
- 平台管理员可以配置模板、风控规则、查看审计日志和处理复核队列。
- 系统不暴露客服和用户真实手机号。
- 系统不允许任意批量短信代发，不支持绕过实名或平台风控。

## 非目标

- 不做非实名号码、虚拟号批量代发或伪装号码。
- 不做开放式群发营销工具。
- 不做完全自由短信内容代发。
- 不在 MVP 中接入彩信 MMS、5G 消息或双向隐私号通话。
- 不允许没有业务授权依据的传话工单。

## 用户角色

### 管理员

- 管理客服账号。
- 配置合规短信服务商。
- 管理传话模板。
- 配置风控规则。
- 查看全部工单、消息、图片、访问记录和审计日志。
- 处理高风险内容复核。

### 客服

- 创建或处理分配给自己的传话工单。
- 选择已启用模板，填写变量和受控补充说明。
- 上传图片附件。
- 提交发送。
- 查看自己工单的发送状态、用户访问状态和失败原因。

### 用户

- 收到合规短信提醒或由客服提供的链接。
- 打开传话页面。
- 输入手机号后四位进行校验。
- 查看文字内容和图片。

## MVP 范围

### 工单

每一次传话必须归属于一个工单。工单记录：

- 业务类型。
- 外部业务 ID，例如订单号、咨询编号或售后单号。
- 用户手机号。
- 用户授权来源。
- 授权时间。
- 联系有效期。
- 创建人。
- 处理客服。
- 工单状态。
- 关闭原因。

没有业务依据和有效期时，不能创建工单。

### 模板和受控补充说明

客服不能发送完全任意短信。客服发送时必须选择已启用模板，并填写模板变量。允许填写一段受控补充说明：

- 字数限制。
- 敏感词检测。
- 发送频率限制。
- 风险命中后进入人工复核。
- 可由管理员按客服账号关闭该能力。

### 图片传话

图片不通过 SMS 直接发送。系统支持图片上传并生成安全访问链接。图片要求：

- 仅允许常见图片格式：jpg、jpeg、png、webp。
- 限制单张大小和每条传话图片数量。
- 保存到现有本地上传目录或腾讯 COS。
- 访问前需要随机 token 和手机号后四位校验。
- 默认 7 天过期。
- 记录访问日志。
- 支持管理员删除或下架图片。

### 短信提醒

MVP 可以接入一家合规短信服务商发送提醒短信。短信只发送审核模板，例如：

> 您有一条来自平台客服的服务消息，请在有效期内查看：{{link}}

短信服务商通过适配层接入，第一版只实现一个真实服务商或 mock provider，但内部接口保持稳定。

### 风控

MVP 采用中等风控：

- 每个客服每小时、每天发送上限。
- 每个用户每天接收上限。
- 夜间发送默认禁止，可由管理员开启。
- 文本敏感词拦截。
- 图片类型、大小、数量限制。
- 同一链接或同一图片重复发送限制。
- 异常频率进入复核队列。
- 复核通过后才能发送提醒。

## 页面设计

### 管理后台

新增后台入口：`/admin/message-relay`

主要页面：

- 工单列表：搜索、筛选状态、查看发送状态和访问状态。
- 工单详情：业务依据、用户脱敏手机号、消息记录、图片、风控记录、访问日志。
- 模板管理：草稿、待审核、已启用、已停用。
- 复核队列：高风险文本、图片或异常发送请求。
- 服务商设置：短信 provider 配置、回调密钥、发送开关。
- 风控设置：频率、夜间发送、图片限制、补充说明开关。

### 客服工作台

新增客服入口可以放在现有用户中心或后台简化视图：

- 我的工单列表。
- 创建传话工单。
- 选择模板。
- 填写变量和补充说明。
- 上传图片。
- 预览用户看到的传话页面。
- 提交发送。
- 查看发送回执和用户访问状态。

### 用户传话页

新增公开页面：

- `/relay/[token]`
- 展示平台名称、消息有效期和校验表单。
- 用户输入手机号后四位。
- 校验通过后展示传话内容和图片。
- 链接过期、已关闭、校验失败时展示明确状态。

## 技术方案

### 架构

沿用当前项目技术栈：

- Next.js App Router。
- TypeScript。
- Prisma。
- PostgreSQL。
- 现有 `User`、`AdminAuditLog`、上传存储和 COS 能力。

新增模块保持小边界：

- `src/lib/message-relay-rules.ts`：纯规则函数，处理状态、有效期、频控、校验。
- `src/lib/message-relay-provider.ts`：短信服务商统一接口。
- `src/lib/message-relay-storage.ts`：图片附件访问 URL 和过期策略。
- `src/app/admin/message-relay/*`：管理员页面。
- `src/app/relay/[token]/page.tsx`：用户公开访问页。
- `src/app/api/message-relay/*`：上传、访问校验、服务商回调。

### 服务商适配层

统一接口：

```ts
type RelaySmsProvider = {
  sendTemplateSms(input: {
    phone: string;
    templateCode: string;
    variables: Record<string, string>;
    outId: string;
  }): Promise<{ providerMessageId: string; raw: unknown }>;

  parseWebhook(input: Request): Promise<{
    providerMessageId: string;
    status: "delivered" | "failed" | "unknown";
    reason?: string;
    raw: unknown;
  }>;
};
```

MVP 可以先实现 mock provider，后续接入阿里云、腾讯云或其他合规短信服务商。

### 数据模型草案

建议新增以下 Prisma 模型：

- `MessageRelayTicket`：传话工单。
- `MessageRelayTemplate`：传话模板。
- `MessageRelayMessage`：一次传话提交。
- `MessageRelayAttachment`：图片附件。
- `MessageRelayDelivery`：短信提醒发送记录。
- `MessageRelayAccessLog`：用户打开和校验记录。
- `MessageRelayReview`：风控复核记录。

关键状态：

- 工单：`open`、`closed`、`expired`。
- 消息：`draft`、`pending_review`、`approved`、`sent`、`failed`、`cancelled`。
- 模板：`draft`、`pending_review`、`active`、`disabled`。
- 发送：`pending`、`sent`、`delivered`、`failed`。

### 核心流程

1. 客服创建工单，填写业务依据、授权来源、手机号和有效期。
2. 系统校验手机号、授权时间和有效期。
3. 客服选择模板，填写变量和补充说明。
4. 客服上传图片。
5. 系统运行风控。
6. 低风险内容直接生成传话 token；高风险内容进入复核。
7. 系统通过合规短信模板发送提醒链接。
8. 用户打开 `/relay/[token]`。
9. 用户输入手机号后四位。
10. 系统校验 token、有效期、工单状态和手机号后四位。
11. 用户查看内容和图片。
12. 系统记录访问日志。

### 安全与合规

- 手机号在页面和日志中默认脱敏展示。
- 公开链接使用高熵随机 token，不使用可猜 ID。
- token 默认 7 天过期。
- 图片不能直接公开枚举访问。
- 管理员操作写入 `AdminAuditLog`。
- 服务商回调需要签名校验或共享密钥校验。
- 所有发送内容、变量、补充说明和附件保存审计副本。
- 删除图片时只删除安全范围内的本地文件或 COS 对象。

## 错误处理

- 工单无授权依据：拒绝创建。
- 工单过期：禁止发送新消息。
- 风控命中：进入复核，不直接发送提醒。
- 服务商发送失败：记录失败原因，允许客服或管理员重试。
- 链接过期：用户页展示过期状态。
- 手机号后四位校验失败：记录失败次数，超过限制后临时锁定该 token。
- 图片不存在或已删除：展示附件不可用，不影响文字内容查看。

## 测试策略

- 单元测试：
  - 工单有效期规则。
  - 手机号脱敏和后四位校验。
  - token 过期规则。
  - 频控规则。
  - 模板变量校验。
  - 风控命中规则。

- 集成测试：
  - 创建工单到发送提醒。
  - 高风险消息进入复核。
  - 用户打开链接并通过校验。
  - 服务商回调更新发送状态。

- 端到端测试：
  - 管理员配置模板。
  - 客服创建工单、上传图片、发送。
  - 用户通过公开页面查看消息。

## 推荐决策

- MVP 先使用 mock provider，把服务商接入留到第二个小版本。
- 夜间默认禁止 22:00 到次日 08:00。
- 图片 7 天后禁止访问，后台保留 30 天审计期后再进入清理任务。
- 客服入口先放在后台，使用现有管理员布局快速落地；后续再拆成独立客服工作台。
