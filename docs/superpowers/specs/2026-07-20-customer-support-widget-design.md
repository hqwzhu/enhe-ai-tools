# 客服悬浮窗口设计

## 状态

方案 A 已由用户确认，本文用于开发前的书面复核。本文只覆盖公开网站客服悬浮窗口，不包含实时聊天、客服工单或 AI 自由问答。

## 背景与目标

ENHE AI 需要在公开网站页面提供一个简单、低打扰的客服入口，帮助访客快速获得官网常见问题的固定答案，并允许访客提交留言。留言提交后通过现有管理员邮件通知能力提醒站长。

目标：

- 让访客在中英文公开页面都能随时找到客服入口。
- 通过固定 FAQ 解答产品、价格、教程和购买流程等通用问题。
- 允许访客不注册、不提供手机号，仅凭问题内容提交留言。
- 将留言发送到现有 `ADMIN_ALERT_EMAILS` 收件人列表。
- 保持实现简单，不增加客服数据库、后台客服中心或第三方客服平台。

## 非目标

- 不提供实时人工聊天或消息来回会话。
- 不接入大模型进行开放式自动回答。
- 不增加用户注册、手机号、文件上传或订单自动查询。
- 不增加客服数据库、站内留言历史和管理员处理状态。
- 不在登录、注册、支付和后台页面展示客服组件。
- 不创建独立的管理员邮箱配置；复用现有 SMTP 和管理员通知配置。

## 用户体验

### 覆盖范围

客服组件由公开站点统一外壳渲染，覆盖中英文公开页面。认证页面、支付页面和后台页面不渲染。

### 入口与关闭

- 入口固定在右下角，使用客服图标和简短的“客服”标签。
- 首次进入页面时默认收起，不自动弹出窗口。
- 点击入口打开客服面板。
- 面板右上角提供关闭按钮。
- 关闭后保留右下角小图标，用户可随时重新打开。
- 页面刷新或跨公开页面跳转不会自动展开窗口。
- 桌面端使用右下角面板；移动端使用接近底部的响应式面板，不能遮挡主要导航和提交按钮。

### 打开后的内容

第一屏只展示欢迎语和固定常见问题按钮，不立即展示自由输入框：

> 你好，我可以帮你快速了解产品、价格、教程和购买流程。

常见问题：

1. ENHE AI 是什么？
2. 如何找到适合我的产品？
3. 产品价格和购买方式是什么？
4. 免费产品和收费产品有什么区别？
5. 购买后如何查看教程或课程内容？
6. AI 账号订购和升级服务是什么？
7. 没有找到答案，提交留言

FAQ 答案由本地固定配置提供。答案可以包含当前语言对应的站内链接，但不得承诺实时订单状态、账号权限或人工处理时效。

### 留言表单

点击“提交留言”后展示表单：

- 问题内容：必填，1 至 2,000 个字符。
- 联系邮箱：可选，填写时必须通过邮箱格式校验。
- 不显示管理员邮箱。
- 不要求登录，不收集手机号。

提交成功后显示“留言已发送，我们会尽快处理”。提交失败时显示明确的重试提示，不显示服务器或 SMTP 内部信息。

## 技术设计

### 组件位置

客服组件挂载在现有公开站点外壳 `src/components/public-site-chrome.tsx` 内，确保公开页面统一显示，并避开认证、支付和后台布局。

建议新增一个客户端组件，例如 `src/components/customer-support-widget.tsx`，负责：

- 展示收起、展开、FAQ 回答和留言表单状态。
- 处理移动端尺寸和键盘焦点。
- 读取当前语言和当前相对路径。
- 提交表单并展示成功、失败、限流状态。

FAQ 文案放在客服组件旁的独立配置模块中，按 `zh` 和 `en` 分组。第一版修改 FAQ 需要代码修改和部署，这是方案 A 接受的维护取舍。

### API

新增 `POST /api/support`，请求体包含：

```ts
{
  message: string;
  email?: string;
  locale: "zh" | "en";
  pagePath?: string;
  website?: string;
}
```

`website` 为隐藏防机器人字段，正常浏览器必须为空。服务端使用 Zod 校验输入，规范化空白字符和路径长度，并拒绝超长或格式非法的数据。

接口只接受 `application/json` 请求，并校验 `Origin`；没有 `Origin` 时校验 `Referer`，拒绝明确来自其他站点的请求。

### 邮件通知

新增客服邮件构造和发送逻辑，复用现有：

- `ADMIN_ALERT_EMAILS` 或 `ADMIN_ALERT_EMAIL` 收件人配置。
- `SMTP_HOST`、`SMTP_PORT`、`SMTP_USER`、`SMTP_PASSWORD` 和 `SMTP_FROM`。
- 现有 SMTP 超时配置和 UTF-8 邮件设置。

邮件主题：

```text
[ENHE AI] 新客服留言 - {pagePath}
```

邮件正文包含：

- 留言内容。
- 用户联系邮箱，如有填写。
- 来源页面路径。
- 当前语言。
- 提交时间。

当用户填写邮箱时，将其设置为邮件 `Reply-To`；未填写时不设置。邮件正文和 HTML 表格中的用户内容必须经过转义。

### 提交流程

1. 客户端校验问题内容和邮箱格式。
2. 客户端提交到 `POST /api/support`。
3. 服务端检查隐藏字段、字段长度、邮箱格式和提交频率。
4. 服务端读取邮件配置；配置不完整时返回可重试错误。
5. 服务端发送邮件到管理员收件人列表。
6. 邮件成功后返回成功状态，客户端清空表单并显示成功提示。

方案 A 不保存留言记录。因此，邮件发送失败时不能宣称留言已收到；接口返回失败，用户可以稍后重试或使用公开联系邮箱。

## 安全与滥用控制

- 使用同源 `POST` 接口，只接受 `application/json`，并校验 `Origin`/`Referer`，不允许跨站任意触发邮件。
- 服务端再次校验所有客户端字段，不能只依赖前端校验。
- 对单个 IP 采用简单内存限流，例如每小时最多 3 次；触发后返回 HTTP 429。
- 使用隐藏字段拦截常见机器人提交。
- 对邮件主题、正文和 `Reply-To` 做格式限制和 HTML 转义。
- 不把 SMTP 配置、管理员邮箱列表或内部异常堆栈返回给浏览器。
- 日志只记录发送结果和必要诊断信息，不记录完整留言正文或用户邮箱。

内存限流会在服务进程重启后清空，并且不适合多实例全局限流；这是方案 A 的已知限制。若后续留言量明显增加，再升级到带持久化记录的方案 B。

## 错误处理

- 缺少问题内容：HTTP 400，提示填写问题。
- 邮箱格式错误：HTTP 400，提示检查邮箱。
- 隐藏字段非空：HTTP 400，不泄露风控细节。
- 提交过于频繁：HTTP 429，提示稍后再试。
- SMTP 配置缺失：HTTP 503，提示暂时无法提交。
- SMTP 发送失败：HTTP 503，提示稍后重试或使用公开联系邮箱。
- 未预期异常：HTTP 500，页面显示通用失败提示，服务端记录错误摘要。

## 验收标准

- 中英文公开页面均显示客服图标。
- 登录、注册、支付和后台页面不显示客服组件。
- 桌面端和移动端面板不遮挡主要操作，关闭后保留可重新打开的小图标。
- FAQ 点击后显示正确语言的固定答案和站内链接。
- 留言内容必填，邮箱可选且格式校验有效。
- 成功提交后管理员邮箱收到完整通知，填写邮箱时可以直接回复用户。
- SMTP 异常、配置缺失和限流场景均不会误报成功。
- 用户输入 HTML、脚本字符或超长内容时不会执行、破坏邮件或绕过校验。
- 组件和 API 有自动化测试，最终通过 `npm test`、`npm run lint`、`npm run typecheck` 和针对公开页面的浏览器验证。
