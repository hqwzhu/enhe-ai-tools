# ENHE API Gateway Acceptance Criteria

Date: 2026-07-05

Status: Phase 1 planning draft

## 验收原则

- 验收以用户可完成真实闭环为准，不只验收页面存在。
- 所有关键判断必须能追溯到阶段 0 审计依据或阶段 1 页面/流程定义。
- 不复制第三方品牌、UI、文案或代码。
- 不读取或输出 `.env`、`zpay.env` 中的真实密钥、Token、数据库连接串。

## 1. 用户与权限

必须通过的验收项：

- 未登录用户访问 `/user/api/*` 会跳转登录，登录后返回原目标页。
- 已登录用户首次进入 `/user/api` 可初始化开发者资料。
- 普通用户只能访问自己的 API Key、日志、余额、订单和推荐数据。
- 冻结 API 用户后，Gateway 拒绝运行时请求，但用户仍可登录查看账单、日志和状态。
- 管理后台 `/admin/api/*` 必须复用管理员权限门禁，依据 `src/app/admin/layout.tsx` 和 `src/lib/auth.ts`。

失败条件：

- 普通用户能访问其他用户 API 数据。
- 冻结状态不影响 Gateway 请求。
- 未登录用户可进入控制台数据页。
- 用户被冻结后无法查看必要账单和日志。

人工测试方法：

- 用未登录浏览器访问 `/user/api`。
- 用两个普通账号互查 Key、日志、账单 URL。
- 将测试用户冻结后继续用旧 Key 请求 Gateway。
- 用普通账号访问 `/admin/api`。

自动化测试建议：

- 为控制台页面加权限集成测试。
- 为 Gateway 开发者状态校验加单元和集成测试。
- 增加跨用户访问回归测试。

## 2. API Key

必须通过的验收项：

- 创建 Key 时只显示一次完整明文。
- 数据库存储 hash、前缀、状态、创建时间、最后使用时间，不存明文。
- Key 列表只展示前缀和元信息。
- 用户可撤销自己的 Key。
- 管理员可撤销任意 Key，但不看到完整明文。
- 撤销后 Gateway 立即拒绝请求。

失败条件：

- 日志、页面或数据库中出现完整 Key 明文。
- 已撤销 Key 仍可调用。
- 用户可撤销他人 Key。
- 创建超过上限未被拦截。

人工测试方法：

- 创建 Key 后刷新页面，确认无法再次查看完整 Key。
- 用撤销前后同一个 Key 调用 `/v1/models`。
- 在日志和后台搜索 Key 明文，确认不出现。

自动化测试建议：

- API Key 生成、hash、验证、撤销单元测试。
- “明文只返回一次”集成测试。
- secret redaction 快照测试。

## 3. OpenAI-compatible Gateway

必须通过的验收项：

- `GET /v1/models` 返回 ENHE 可用模型列表。
- `POST /v1/chat/completions` 支持基础请求。
- `stream=false` 返回完整 JSON。
- `stream=true` 返回基础流式响应。
- 鉴权失败返回 `401`。
- 余额不足返回 `402`，且不调用上游。
- 模型关闭返回明确错误。
- 成功请求写入日志和扣费流水。

失败条件：

- 响应格式无法被常见 OpenAI-compatible 客户端解析。
- 余额不足仍产生上游调用。
- 失败请求错误体泄露上游密钥或内部栈。
- `stream=true` 无日志或无扣费策略。

人工测试方法：

- 使用测试 Key 调用 `/v1/models`。
- 分别发起 `stream=false` 和 `stream=true` chat 请求。
- 将余额调低到不足后再次请求。
- 关闭一个模型后请求该模型。

自动化测试建议：

- 合约测试覆盖 `/v1/models` 和 `/v1/chat/completions`。
- mock 上游供应商，验证请求转发和错误映射。
- 余额不足测试断言上游 mock 未被调用。
- 流式响应测试验证事件格式和日志落库。

## 4. Anthropic-compatible Gateway

必须通过的验收项：

- `POST /v1/messages` 支持基础 Anthropic-compatible 请求。
- 支持 `stream=false` 和 `stream=true`。
- 鉴权、限流、余额、模型状态规则与 OpenAI-compatible Gateway 一致。
- 日志可区分 `/v1/messages` 请求。
- 错误响应使用 ENHE 自有错误码与说明。

失败条件：

- Claude Code 配置后无法发起测试请求。
- `/v1/messages` 与 `/v1/chat/completions` 扣费规则不一致且无说明。
- 流式中断后没有明确日志状态。

人工测试方法：

- 按 `/user/api/docs` 中 Claude Code 文档配置测试请求。
- 分别测试正常、余额不足、模型关闭、Key 撤销。
- 在日志页按路径筛选 `/v1/messages`。

自动化测试建议：

- Anthropic-compatible request/response 合约测试。
- 流式事件测试。
- 错误映射测试。

## 5. 日志

必须通过的验收项：

- 每个 Gateway 请求有 request id。
- 日志至少包含用户、Key 前缀、路径、模型、状态、token、延迟、错误码、时间。
- 用户只能看自己的日志。
- 管理员可按用户、Key 前缀、模型、状态、request id 查询。
- 余额不足、限流、鉴权失败、模型关闭、上游失败均有明确状态。
- 默认不保存 prompt/completion 正文。

失败条件：

- 日志缺失导致无法排查扣费。
- 日志包含完整 API Key。
- 用户可看到其他用户日志。
- 失败请求没有错误码。

人工测试方法：

- 发起成功、余额不足、限流、撤销 Key 请求。
- 在用户日志页和管理员日志页分别查询。
- 检查日志详情是否含完整 Key 或敏感配置。

自动化测试建议：

- Gateway 请求日志集成测试。
- 日志权限测试。
- 敏感字段扫描测试。

## 6. 额度与扣费

必须通过的验收项：

- 用户有 API 钱包余额。
- 额度来源可区分充值、套餐、赠送、推荐、管理员调整。
- 每次余额变化写入不可变流水。
- 成功可计费请求关联扣费流水和请求日志。
- 余额不足请求不扣费、不调用上游。
- 并发请求不能让余额变为负数。

失败条件：

- 余额与流水汇总不一致。
- 并发请求产生负余额。
- 同一请求重复扣费。
- 支付或管理员调整没有流水。

人工测试方法：

- 给测试用户发放额度并发起请求。
- 检查余额、扣费流水、日志关联。
- 模拟并发请求和低余额场景。
- 做一次管理员手动加减余额。

自动化测试建议：

- 钱包流水单元测试。
- 并发扣费集成测试。
- 幂等扣费测试。
- Decimal 精度测试。

## 7. 套餐与支付

必须通过的验收项：

- `/ai-api/pricing` 和 `/user/api/billing` 展示一致套餐。
- 用户可购买套餐或由管理员手动开通。
- 支付成功后额度幂等发放到 API 钱包。
- 支付处理中有明确状态。
- 退款/失效规则链接到法律条款。

失败条件：

- 支付成功但无额度流水。
- 重复回调导致重复发放。
- 套餐页承诺未实现能力。
- 订单金额与发放额度无法追踪。

人工测试方法：

- 创建测试订单并完成支付或模拟管理员手动开通。
- 刷新账单页和用量页。
- 重复触发回调或补发操作，确认不重复发放。

自动化测试建议：

- 支付回调幂等测试。
- 订单到钱包发放集成测试。
- 套餐配置一致性测试。

## 8. 推荐系统

必须通过的验收项：

- 用户可复制邀请链接。
- 好友注册后记录推荐关系。
- 好友完成验证和首次有效 API 调用后才发放基础奖励。
- 奖励进入额度流水。
- 管理员可查看、冻结或标记异常推荐。

失败条件：

- 仅注册就发放完整奖励。
- 同一好友重复产生奖励。
- 奖励没有流水记录。
- 用户可伪造邀请关系。

人工测试方法：

- 用邀请链接注册测试好友。
- 未调用 API 前检查奖励状态。
- 完成首次有效调用后检查双方奖励。
- 管理员冻结异常推荐。

自动化测试建议：

- 推荐关系唯一性测试。
- 奖励触发条件测试。
- 反重复发放测试。

## 9. 文档中心

必须通过的验收项：

- `/ai-api/docs` 提供公开快速开始、接口说明、错误码和 Key 安全说明。
- `/user/api/docs` 提供用户上下文配置入口，但不展示完整 Key。
- Codex 配置文档可完成测试请求。
- Claude Code 配置文档可完成测试请求。
- 文档示例使用 ENHE Base URL 和 ENHE 自有文案。

失败条件：

- 文档包含真实密钥或环境文件值。
- 文档复制第三方品牌文案。
- 示例路径与 Gateway 实际接口不一致。
- 错误码无法指导用户排查。

人工测试方法：

- 按 Codex 文档从零配置测试。
- 按 Claude Code 文档从零配置测试。
- 在文档中搜索敏感字符串和第三方品牌复制痕迹。

自动化测试建议：

- 文档链接检查。
- 示例命令 smoke test。
- 文档 secret-pattern 扫描。

## 10. 管理后台

必须通过的验收项：

- `/admin/api` 提供 API 业务总览。
- 管理员可冻结用户、撤销 Key、查询日志、调整余额、关闭模型、查看订单和推荐。
- 冻结、撤销、调余额、关模型、补发额度均写审计日志。
- 后台不显示完整 API Key 和上游供应商密钥。

失败条件：

- 普通用户能访问后台。
- 管理员操作无审计。
- 调余额没有原因或流水。
- 关闭模型后 Gateway 仍可调用。

人工测试方法：

- 用管理员账号执行每个关键操作。
- 用普通账号访问后台 URL。
- 检查审计日志详情。

自动化测试建议：

- 后台权限测试。
- 管理员操作审计测试。
- 模型关闭到 Gateway 生效集成测试。

## 11. 安全与合规

必须通过的验收项：

- `.env`、`zpay.env` 真实值不进入代码、文档、日志或响应。
- 上游供应商密钥只在服务端使用。
- API Key、用户数据、日志权限有隔离。
- API 服务条款、隐私政策、退款规则、禁止用途已扩展。
- 请求内容默认不保存正文。
- 第三方供应商条款已确认允许对应代理/转发/商业使用。

失败条件：

- 文档或日志出现真实 Token、密钥或数据库连接串。
- 前端可见上游 provider key。
- 用户协议未覆盖 API 服务。
- 未确认供应商条款就公开收费。

人工测试方法：

- 对 `docs/enhe-api`、控制台页面和 Gateway 日志做敏感模式扫描。
- 检查法律页面是否包含 API 条款。
- 检查浏览器网络面板是否暴露服务端密钥。

自动化测试建议：

- secret scan。
- 权限隔离测试。
- 安全响应头和错误脱敏测试。

## 12. 灰度上线

必须通过的验收项：

- 支持内部账号或白名单灰度。
- 支持关闭模型或暂停 Gateway 新请求。
- 支持低额度、低并发、低速率默认策略。
- 支持基础监控：成功率、错误率、余额不足、上游失败、成本。
- 灰度期间可回滚公开入口或隐藏购买入口。

失败条件：

- 所有用户一次性开放且无关闭开关。
- 模型故障无法快速关闭。
- 公开收费前无日志和扣费验证。
- 无法区分 Gateway 问题和上游供应商问题。

人工测试方法：

- 只对白名单用户开放控制台。
- 关闭某个模型并验证用户请求失败原因。
- 暂停购买入口但保留已购买用户使用。
- 查看灰度指标。

自动化测试建议：

- feature flag/白名单测试。
- 模型关闭开关测试。
- Gateway 健康检查和告警 smoke test。
