# ENHE API Gateway Phase 2 Scope

Date: 2026-07-05

Status: Phase 1 planning draft

## Phase 2 定位

第二阶段在 MVP 可用闭环稳定后推进，重点从“能安全计费和调用”扩展到“更高可用性、更细运营、更强工具文档、更完整商业化”。

Phase 2 不应在 MVP 账本、日志、API Key、安全、法律和灰度能力未验证前提前实现。

## 第二阶段功能

| 序号 | 功能 | 目标 | 依赖 |
| --- | --- | --- | --- |
| 1 | `/v1/responses` | 增加 OpenAI Responses API 兼容入口，覆盖新一代工具和应用接入。 | MVP Gateway 合约测试、模型路由稳定 |
| 2 | 多供应商 fallback | 上游失败时可切换到备选供应商或模型。 | Provider 合规确认、成本策略、幂等日志 |
| 3 | 上游失败重试 | 对可重试错误执行有限重试，降低偶发失败。 | 请求幂等、流式失败策略、上游错误分类 |
| 4 | 服务状态页 | `/ai-api/status` 展示接口、模型、事故和维护状态。 | 健康检查、事件记录、模型状态数据 |
| 5 | 成本告警 | 对用户余额、异常消耗、管理员成本和供应商成本发出告警。 | 钱包流水、日志聚合、通知通道 |
| 6 | 用量日报 | 给用户或管理员生成每日调用、消耗、错误和余额报告。 | 日志聚合、邮件/站内通知 |
| 7 | Cursor/Cline/ChatBox 深度文档 | 增加更多工具的 ENHE 配置指南和排障步骤。 | MVP 文档中心、实际工具验证 |
| 8 | 模型分级和套餐权限 | 不同套餐可用不同模型、上下文、速率和并发。 | API 计划模型、Gateway 权限判断 |
| 9 | 充值自动回调 | API 套餐/充值支付成功后自动幂等发放额度。 | 现有支付复用方案、API 钱包流水 |
| 10 | 推荐系统反作弊增强 | 增加设备/IP hash/支付/使用行为等风险判断和人工审核。 | MVP 推荐关系、风控数据 |
| 11 | 管理后台数据看板 | 增加收入、成本、毛利、模型失败率、活跃用户、留存指标。 | 日志聚合、成本数据、订单数据 |
| 12 | API 调用内容隐私开关 | 用户可选择是否允许保存请求/响应内容用于排障。 | 隐私政策、数据保留、权限隔离 |
| 13 | 团队空间 | 支持组织、成员、角色、共享余额、团队 Key。 | 用户权限模型、账单分摊、审计 |
| 14 | 企业套餐 | 企业额度、合同、专属模型权限、SLA、人工支持。 | 团队空间、财务流程、法务条款 |

## Phase 2 评估标准

- MVP 请求成功率、扣费一致性和余额安全已经通过灰度验证。
- 用户真实需求证明某项能力能提升留存、付费或支持效率。
- 每项能力都有明确的数据模型、权限边界、失败策略和验收标准。
- 不因 Phase 2 扩展破坏 MVP 的 API Key、安全、日志、扣费和法律闭环。

## Phase 2 优先级建议

第一优先级：

- 充值自动回调。
- 成本告警。
- 服务状态页。
- 上游失败重试。

第二优先级：

- 多供应商 fallback。
- Cursor/Cline/ChatBox 深度文档。
- 模型分级和套餐权限。
- 管理后台数据看板。

第三优先级：

- `/v1/responses`。
- API 调用内容隐私开关。
- 推荐系统反作弊增强。
- 团队空间和企业套餐。

## Phase 2 风险提示

- fallback 和重试可能导致重复扣费或重复日志，必须先定义幂等策略。
- 内容隐私开关涉及 prompt/completion 存储，必须先完成法律条款和保留策略。
- 团队空间会显著扩大权限模型，不应混入 MVP。
- 企业套餐会引入合同、SLA、发票、专属支持，不应在消费级 MVP 中提前承诺。
