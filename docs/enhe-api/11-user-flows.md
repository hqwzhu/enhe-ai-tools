# ENHE API Gateway User Flows

Date: 2026-07-05

Status: Phase 1 planning draft

## 阶段 0 依据

这些用户路径基于阶段 0 的混合架构结论：

- 登录、用户中心、支付、后台、法律由现有 ENHE 网站承载，依据 `src/lib/auth.ts`、`src/app/user/*`、`src/app/admin/*`、`src/lib/zpay-orders.ts`、`src/lib/legal.ts`。
- `/v1/*` 运行时由独立 Gateway 服务承载，依据 `docs/enhe-api/02-architecture-options.md`。
- API Key、钱包、日志、模型路由、推荐奖励是新增能力，依据 `docs/enhe-api/01-product-scope.md` 和 `docs/enhe-api/03-risk-register.md`。

## 1. 新用户首次使用路径

目标：让从公开页进入的新用户完成注册、创建 API Key、配置工具、发起首次请求并看到日志和余额。

路径：

1. 用户访问 `/ai-api`。
2. 用户点击开始使用，进入注册/登录。
3. 注册或登录成功后进入 `/user/api`。
4. 系统检测是否存在开发者资料；若不存在，展示初始化表单。
5. 用户确认开发者资料，系统创建开发者资料并进入控制台。
6. 用户在 `/user/api/keys` 创建 API Key。
7. 系统只在创建成功时展示一次完整 API Key，并提示安全保存。
8. 用户进入 `/user/api/docs` 或 `/ai-api/docs` 阅读快速开始。
9. 用户选择 Codex 配置文档，复制 Base URL、模型名和 API Key。
10. 用户在本地工具中发起首次请求。
11. Gateway 校验 API Key、额度、限流和模型路由。
12. Gateway 调用上游并写入请求日志与扣费流水。
13. 用户回到 `/user/api/logs` 查看首次请求记录。
14. 用户进入 `/user/api/usage` 查看余额和消耗。

关键状态：

- 未登录：跳转登录，登录后回到目标页。
- 无开发者资料：必须先初始化。
- 无 API Key：展示创建入口和空状态。
- 无余额：允许创建 Key 和阅读文档，但运行时请求返回 `402`。

验收要点：

- 从 `/ai-api` 到 `/user/api` 的登录后回跳可用。
- API Key 明文只显示一次。
- 首次请求日志在用户控制台可见。
- 余额变化可追踪到扣费流水。

## 2. 老用户开通 API 路径

目标：让已有 ENHE 用户不重复注册，直接在用户中心开通 API 服务。

路径：

1. 用户登录 ENHE AI。
2. 用户进入现有用户中心。
3. 用户点击 ENHE API 入口，进入 `/user/api`。
4. 系统检查开发者资料，不存在则引导初始化。
5. 用户创建 API Key。
6. 若用户已有赠送额度，展示可用余额。
7. 若余额不足或无套餐，用户进入 `/user/api/billing`。
8. 用户购买套餐、充值额度，或等待管理员手动开通套餐。
9. 支付成功或手动开通后，钱包余额更新。
10. 用户开始使用 API。

关键状态：

- 已登录但未初始化开发者资料：进入初始化。
- 已有未撤销 Key：展示 Key 列表，不强迫新建。
- 支付处理中：展示订单状态和刷新入口。

验收要点：

- 现有 ENHE 账号可直接开通 API。
- API 功能入口不破坏当前 `/user` 页面既有订单、下载、设置能力。
- 购买或手动开通后的额度进入 API 钱包，不只停留在订单状态。

## 3. Codex 使用路径

目标：让用户按 ENHE 文档配置 Codex 并验证请求日志。

路径：

1. 用户进入 `/user/api/keys` 创建 API Key。
2. 用户进入 `/user/api/docs` 的 Codex 配置文档。
3. 文档展示 ENHE API Base URL、认证方式、推荐模型名和测试命令。
4. 用户在 Codex 配置中填入 Base URL 和 API Key。
5. 用户发起测试请求。
6. Gateway 处理 `GET /v1/models` 或 `POST /v1/chat/completions`。
7. 用户回到 `/user/api/logs`，按时间、模型、状态筛选。
8. 用户确认测试请求成功，查看消耗和余额。

关键状态：

- Key 未保存：文档页提示先创建 Key。
- Base URL 配错：请求日志可能无记录，文档提供本地排查项。
- 认证失败：Gateway 返回 `401`，文档说明检查 Key 是否撤销或粘贴错误。

验收要点：

- Codex 文档不复用第三方站点文案，使用 ENHE 自有说明。
- 配置项只包含 ENHE Base URL、ENHE API Key 和模型名。
- 测试请求能在日志页验证。

## 4. Claude Code 使用路径

目标：让用户使用 Anthropic-compatible endpoint 完成 Claude Code 兼容配置。

路径：

1. 用户创建 API Key。
2. 用户进入 Claude Code 配置文档。
3. 文档展示 Anthropic-compatible endpoint，对应 `POST /v1/messages`。
4. 用户配置 Base URL、API Key 和模型名。
5. 用户发起测试请求。
6. Gateway 校验请求格式并转发到对应上游模型。
7. Gateway 返回 Anthropic-compatible 响应。
8. 用户在 `/user/api/logs` 查看请求路径、模型、状态和扣费。

关键状态：

- 请求格式不兼容：返回 `400` 并给出 ENHE 错误码。
- 模型未启用：返回 `404` 或 `403`，日志记录模型关闭原因。
- 余额不足：返回 `402`，用户进入充值路径。

验收要点：

- `/v1/messages` 支持 `stream=false` 和 `stream=true` 的基础路径。
- 日志可区分 OpenAI-compatible 与 Anthropic-compatible 请求。
- 错误响应不泄露上游供应商密钥、原始内部栈或敏感配置。

## 5. 余额不足路径

目标：让余额不足时不产生上游成本，并引导用户恢复服务。

路径：

1. 用户使用 API Key 发起请求。
2. Gateway 在调用上游前检查可用余额。
3. 余额不足时，Gateway 不调用上游。
4. Gateway 返回 `402`，错误体包含 ENHE 错误码、简短原因和充值入口。
5. Gateway 写入一条失败请求日志，标记为余额不足且不扣费。
6. 用户进入 `/user/api/usage` 查看余额。
7. 用户进入 `/user/api/billing` 充值或购买套餐。
8. 支付成功或管理员开通后，钱包余额更新。
9. 用户重新发起请求。

关键状态：

- 余额不足失败日志必须可见。
- 失败日志不应产生扣费流水。
- 如果支付回调延迟，用户看到订单处理中状态。

验收要点：

- 余额不足请求不会调用上游供应商。
- 用户能从错误说明进入充值或套餐页。
- 充值后重新请求可以成功。

## 6. API Key 泄露处理路径

目标：让用户发现异常消耗后能快速排查、撤销和恢复。

路径：

1. 用户在 `/user/api/usage` 发现余额异常下降。
2. 用户进入 `/user/api/logs`，按 API Key 前缀、模型、时间、IP hash、状态筛选。
3. 用户定位可疑 Key。
4. 用户进入 `/user/api/keys` 撤销该 Key。
5. Gateway 立即拒绝已撤销 Key。
6. 用户创建新 Key。
7. 用户更新本地工具配置。
8. 用户可向管理员提交风险排查请求。
9. 管理员在 `/admin/api/logs` 和 `/admin/api/keys` 查看异常。
10. 管理员可冻结用户、调整限流、关闭模型或调整余额，并写入审计日志。

关键状态：

- 撤销操作需二次确认。
- 撤销后不能恢复，只能新建。
- 日志页只显示 Key 前缀，不显示完整 Key。

验收要点：

- 撤销后的 Key 立即失效。
- 用户可用日志完成基本排查。
- 管理员风控动作有审计记录。

## 7. 推荐奖励路径

目标：让推荐奖励以真实使用为触发条件，降低注册刷量。

路径：

1. 用户进入 `/user/api/referrals`。
2. 用户复制 ENHE 邀请链接。
3. 好友通过邀请链接访问 `/ai-api` 并注册。
4. 系统记录推荐关系，但不立即发放全部奖励。
5. 好友完成必要验证。
6. 好友创建 API Key。
7. 好友发起首次有效 API 调用。
8. 系统向邀请人和好友发放基础额度奖励。
9. 双方可在推荐页和额度流水中看到奖励来源。
10. 管理员可在 `/admin/api/referrals` 查看异常推荐关系。

关键状态：

- 同设备、同 IP hash、同支付标识等风险信号进入待审核。
- 好友未验证或未有效调用时，奖励显示为待激活。
- 奖励额度可设置有效期和模型范围。

验收要点：

- 推荐奖励不因注册瞬间直接发放完整额度。
- 奖励发放有对应额度流水。
- 管理员能冻结异常推荐链路。
