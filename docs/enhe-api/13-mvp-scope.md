# ENHE API Gateway MVP Scope

Date: 2026-07-05

Status: Phase 1 planning draft

## 范围原则

MVP 目标是完成 ENHE API Gateway 的可用闭环：公开介绍、登录开通、API Key、兼容接口、流式基础支持、请求日志、额度钱包、扣费、套餐、推荐奖励、后台风控、法律条款和灰度发布。

阶段 0 约束继续有效：

- 复用现有 ENHE 网站承载账号、控制台、支付、后台、法律和文档。
- 独立 Gateway 服务承载 `/v1/*` 运行时。
- 不复制第三方品牌、UI、文案或代码。

## MVP 必须包含

| 序号 | 功能 | MVP 要求 | 主要承载方 |
| --- | --- | --- | --- |
| 1 | 公开介绍页 | `/ai-api` 介绍 ENHE API、适用场景、接口能力和开通入口。 | 现有 ENHE 网站 |
| 2 | 公开套餐页 | `/ai-api/pricing` 展示套餐、额度、试用和退款入口。 | 现有 ENHE 网站 |
| 3 | 用户 API 控制台首页 | `/user/api` 汇总余额、Key、日志、文档和开通状态。 | 现有 ENHE 网站 |
| 4 | 开发者资料初始化 | 首次进入控制台创建 API 开发者资料。 | 现有 ENHE 网站 |
| 5 | API Key 创建、列表、撤销 | Key 明文只展示一次，列表只显示前缀，支持撤销。 | 网站控制台 + Gateway 鉴权 |
| 6 | OpenAI-compatible `/v1/models` | 返回可用模型列表和基础状态。 | 独立 Gateway |
| 7 | OpenAI-compatible `/v1/chat/completions` | 支持基础 chat completions 请求和响应。 | 独立 Gateway |
| 8 | Anthropic-compatible `/v1/messages` | 支持基础 messages 请求和响应。 | 独立 Gateway |
| 9 | `stream=false` 和 `stream=true` 基础支持 | 非流式和流式均能返回、记录日志、按规则扣费。 | 独立 Gateway |
| 10 | 模型路由 | 管理员配置公开模型名到上游模型的基础映射。 | Gateway + 管理后台 |
| 11 | 请求日志 | 记录 request id、用户、Key 前缀、模型、状态、token、延迟、错误。 | Gateway 写入，网站读取 |
| 12 | 额度钱包 | 用户有 API 余额，区分充值、套餐、赠送、推荐、调整来源。 | 网站业务域 + Gateway 扣费 |
| 13 | 扣费流水 | 所有余额变化进入不可变流水，并可关联请求日志或订单。 | 共享业务域 |
| 14 | 余额不足拦截 | 余额不足时 Gateway 返回 `402`，不调用上游，不扣费。 | 独立 Gateway |
| 15 | 基础限流 | 按用户、Key、IP hash、模型或接口设置基础限流。 | 独立 Gateway |
| 16 | 用户用量页 | `/user/api/usage` 展示余额、消耗、趋势和流水。 | 现有 ENHE 网站 |
| 17 | 套餐与账单页 | `/user/api/billing` 展示套餐、订单、支付状态和账单。 | 现有 ENHE 网站 |
| 18 | 手动开通套餐或复用现有支付系统 | MVP 至少支持管理员手动开通；可复用现有支付系统完成自动购买。 | 网站后台 + 支付域 |
| 19 | 推荐奖励基础版 | 邀请链接、好友注册、验证、首次有效调用后发放基础额度。 | 现有 ENHE 网站 |
| 20 | 配置文档中心 | `/ai-api/docs` 和 `/user/api/docs` 提供 Codex、Claude Code、接口和错误码文档。 | 现有 ENHE 网站 |
| 21 | 管理员冻结用户 | 冻结 API 能力，Gateway 拒绝运行时请求，写审计。 | 管理后台 + Gateway |
| 22 | 管理员调整余额 | 支持增加/扣减余额，必须填写原因并写入流水和审计。 | 管理后台 |
| 23 | 管理员关闭模型 | 关闭模型后 Gateway 返回明确错误，文档和列表同步状态。 | 管理后台 + Gateway |
| 24 | 法律条款扩展 | 更新 API 服务条款、隐私、退款、禁止用途和上游依赖说明。 | 现有 ENHE 网站 |
| 25 | 灰度发布 | 内部账号、白名单或小流量逐步开放，具备回滚/关闭模型能力。 | 网站 + Gateway + 运维 |

## MVP 不做

| 序号 | 不做项 | 原因 |
| --- | --- | --- |
| 1 | 复杂模型自动调度 | 先保证基础路由、日志和扣费正确。 |
| 2 | 多供应商智能路由 | 涉及成本、质量、合规和失败策略，放到 Phase 2。 |
| 3 | 高级缓存计费 | 缓存命中、读写 token 和计费规则复杂，MVP 不引入。 |
| 4 | 企业团队空间 | 需要组织、角色、成员账单和权限模型。 |
| 5 | 自研 SDK | MVP 先用标准 HTTP 与工具配置文档满足接入。 |
| 6 | Playground | 会扩大 UI、运行成本和内容安全范围。 |
| 7 | 模型排行榜 | 容易引入质量评估、营销合规和维护成本。 |
| 8 | 自动发票 | 财务流程复杂，先保留订单和账单记录。 |
| 9 | 完整服务状态页 | MVP 可提供基础状态说明，完整事件系统放 Phase 2。 |
| 10 | 高级推荐返佣系统 | MVP 只做基础额度奖励，防止刷量和返佣争议。 |

## MVP 发布门槛

- API Key hash 存储、一次性明文展示、撤销立即生效。
- `/v1/models`、`/v1/chat/completions`、`/v1/messages` 合约测试通过。
- `stream=false` 和 `stream=true` 基础测试通过。
- 请求日志、扣费流水、余额一致性测试通过。
- 并发请求不能造成负余额。
- 余额不足不调用上游。
- 管理员冻结用户、调整余额、关闭模型均写审计日志。
- 法律条款扩展完成。
- 灰度名单和关闭开关可用。

## MVP 关键依赖

- Phase 2 需要先完成数据模型与接口契约设计。
- Gateway 服务部署方案需要明确域名、鉴权、限流存储和密钥管理。
- 支付复用方案需要明确 API 订单到钱包额度的幂等发放机制。
- 法律和供应商条款需要在公开收费前确认。
